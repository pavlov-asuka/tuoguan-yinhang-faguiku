# AGENTS.md

本项目是“中国公募基金托管法规知识库”，视角为商业银行开展公募基金托管业务。

## 工作原则

- 回答法规问题时，优先检索本项目下的本地法规库，而不是凭记忆回答。
- 可以结合联网搜索结果补充核验，尤其是现行性、最新修订、官方来源变动等信息；回答时应区分本地库依据与联网检索结果。
- `03_元数据台账/search.db` 是可重建的检索加速层，不作为原文依据；结论仍应回到台账、文本、原文和官方来源核验。
- 回答时尽量给出法规名称、条款位置、文件路径和官方来源 URL。
- 回答问题时，如发现结论需要依赖但本地库未收录的法规、规则、配套文件或附件，应登记到 `99_unresolved/unresolved.md`，并在回答中说明“本地库暂未收录，已登记到 unresolved.md”，同时区分：
  - 已联网定位到官方来源但尚未入库；
  - 仅有线索、尚待核验；
  - 可先作为辅助资料但不能作为现行正式依据。
- 对上述缺口，应在 `99_unresolved/unresolved.md` 中标注“待月度巡检”，供月度巡检时统一核验和解决。
- 对 `待核验`、`待扩展`、`历史失效`、`辅助资料` 状态的文件，必须明确说明状态，不得当作现行正式依据使用。
- 不要把征求意见稿、修订说明、新闻稿、解读文章当作现行规则。
- 不要把行业手册或操作实务手册当作强制性法规依据。
- 对法规理解可以做业务分析，但要区分“原文依据”和“分析判断”，不得表述为正式法律意见。

## 推荐查询链路

1. 用索引脚本快速召回候选条款；如 `03_元数据台账/search.db` 已存在，优先使用：

   ```powershell
   python tools/search_kb.py "关键词" --limit 8
   ```

2. 用 `03_元数据台账/rules_index.csv` 或 `rules_index.json` 判断命中结果的法规名称、效力状态、优先级、官方来源 URL 和本地路径。
3. 对业务场景型问题，补查 `04_托管业务专题地图/托管业务专题地图.md`，避免只按关键词遗漏相关规则组。
4. 用 `rg` 回看文本上下文或做补充交叉检索，避免把 PowerShell `Select-String` 作为首选全文扫描方式：

   ```powershell
   rg -n "关键词1|关键词2" 02_文本抽取库 03_元数据台账 04_托管业务专题地图
   ```

5. 回到 `01_法规原文库/` 和官方 URL 核验正文、附件、条款位置和现行性。
6. 如发现本地库缺口，登记到 `99_unresolved/unresolved.md`，并按“待月度巡检”标注。

## 文件维护

- 新增法规、替换法规版本、修正元数据、补充官方来源、更新现行状态、扩展专题标签时，必须同步更新相关索引和 Markdown 文件。
- 法规库构建或文本抽取发生变化后，应同步重建 `03_元数据台账/search.db`；该文件是派生索引，可删除后用 `python tools/build_search_index.py` 重建。
- 至少检查并按需更新：
  - `README.md`
  - `00_入口与索引/总目录.html`
  - `00_入口与索引/更新记录.md`
  - `03_元数据台账/rules_index.md`
  - `03_元数据台账/rules_index.csv`
  - `03_元数据台账/rules_index.json`
  - `03_元数据台账/search.db`
  - `04_托管业务专题地图/托管业务专题地图.md`
  - `99_unresolved/unresolved.md`
- 原文文件放入 `01_法规原文库/`。
- 可检索文本放入 `02_文本抽取库/`。
- 无法确认官方正文、附件或现行状态的项目，放入 `99_unresolved/unresolved.md`。
- 已解决的 unresolved 项目，应从 `unresolved.md` 移除或改为“已解决”说明，并同步到台账。
- 每次更改文件后，应提示用户是否需要提交 Git。
